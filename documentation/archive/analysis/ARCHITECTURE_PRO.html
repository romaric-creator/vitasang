📚 Justifications Technologiques

🔴 Pourquoi Redis + BullMQ ?

Problème: Les notifications push bloquaient les requêtes utilisateur → response time lente (3-5s)

Solution: Traitement asynchrone via job queue

Bénéfices:
  • ✓ Response time: 200-500ms (vs 3-5s)
  • ✓ Retry automatique (5 tentatives)
  • ✓ Reliability: Dead Letter Queue
  • ✓ Scalabilité: Multiple workers parallèles

🟡 Pourquoi Sequelize + MySQL ?

Choix: ORM Node.js pour MySQL

Raisons:
  • ✓ Relations complexes (Users → Appointments → Alerts)
  • ✓ Migrations de schéma versionnées
  • ✓ Transactions ACID pour opérations critiques
  • ✓ Indexation automatique (performance)

🟢 Pourquoi Redis Cache ?

Problème: Requêtes répétitives surchargeaient la DB

Solution: Cache distribué avec TTL intelligent

Résultats:
  • ✓ Retrieval: <1ms vs 50-100ms en DB
  • ✓ Réduction de charge DB: 60-80%
  • ✓ TTL adapté par type: User Profile (1h), Appointments (30m), Alerts (5m)
  • ✓ Invalidation intelligente on write

📊 Architecture des Données

Tables Principales (Sequelize ORM)

👥 users

  • id (UUID PK)
  • email (UNIQUE)
  • password (HASHED)
  • bloodType (ENUM)
  • eligibleForDonation

📍 appointments

  • id (UUID PK)
  • userId (FK)
  • appointmentDate
  • status (ENUM)
  • centerId (FK)

🩸 alerts

  • id (UUID PK)
  • creatorId (FK)
  • bloodTypeNeeded
  • quantity (ml)
  • urgencyLevel

💬 messages

  • id (UUID PK)
  • senderId (FK)
  • recipientId (FK)
  • threadId
  • content (TEXT)

Relations & Index Strategy

Relations: (1:N) User → Appointments, Alerts, Messages
Indexes: PRIMARY on id • UNIQUE on users.email • COMPOSITE on (userId, status)
Effect: Requêtes 100-1000x plus rapides

Redis Cache Strategy

Clé Redis Contenu TTL Invalidation
user:{userId}:profile Profil utilisateur 1 heure On update profile
user:{userId}:appointments Liste rendez-vous 30 minutes On create/cancel appt
alert:{alertId}:responses Réponses à alerte 5 minutes On new response
session:{sessionId} Données session 24 heures On logout

Performance Metrics

💨 Sans Cache

  • DB Query: 50-100ms
  • API Response: 200-500ms
  • Peak Load: 100 req/s
  • DB CPU: 80-90%

⚡ Avec Cache Redis

  • Cache Hit: <1ms
  • API Response: 50-100ms
  • Peak Load: 1000 req/s
  • DB CPU: 10-20%
